## Problem Statement

Fintrack is a "gym app for your money" PWA targeting Indonesian fresh workers (first-time
salaried workers, Rp 8–10 jt range). The PRD (full_doc.html) and 16-phase Go backend
plan (docs/superpowers/plans/2026-04-29-backend-mvp.md) are extensively documented but
pre-development.

**Real goal** (refined mid-session): Fintrack is a portfolio piece that demonstrates
**AI-orchestrated development** — the ability to drive AI tooling (Claude Code,
agent workflows, planning skills) to produce a non-trivial, production-quality product.
The Go code is the *output*. The *process* — how a solo builder + AI shipped 25
endpoints, multi-tenant security, AI integrations, and a full PRD — is the actual
showcase.

There is no interview deadline. Build cadence is slow-burn — the artifact should
still read well in 12 months when a hiring manager or recruiter looks at it.

## Portfolio Value (what makes this stand out to a reviewer)

The artifact is not just code. It is the **full AI-build pipeline made visible**:

1. **The 92KB PRD (`full_doc.html`)** — evidence that you can drive AI to produce
   product documentation at a level most solo builders never reach. Indonesian-native,
   styled, with full DB schema, API contract, user stories.
2. **The 16-phase implementation plan** — evidence of structured AI-driven planning.
   Each phase has files, steps, test strategies. This file alone is a portfolio asset.
3. **The office-hours session output (this design doc)** — evidence of using AI for
   *judgment*, not just execution. The premise challenges, the reframes, the
   sequencing decision — these document how you think with AI assistance.
4. **The shipped Go backend** — evidence that the planning actually produces working
   code. 25 endpoints, RLS multi-tenancy, AES income encryption, AI receipt scan,
   weekly narrative cron, debt snowball/avalanche, BYOA tokens.
5. **The README + workflow notes** — explains the build process for a reviewer who
   wants to understand HOW it was made, not just WHAT was made.

The remaining CRUD endpoints prove that the workflow can handle table-stakes work
at quality. The differentiated parts (AI receipt scan, budget engine, RLS, AES)
prove the workflow can handle hard problems.

**What a hiring manager sees in 5 minutes:**

- README lead: "Built solo with Claude Code in ~N weeks. 25 endpoints, AI receipt
  scan, multi-tenant Postgres with RLS, weekly AI-narrated reports."

## Target Audience for the Portfolio Piece

Not "Indonesian fresh workers." The audience is:
- **Future hiring managers and recruiters** who will look at your portfolio when
  you eventually do interview — not today, but some unknown date in the next
  6–18 months.
- **AI-curious tech leaders** evaluating whether you can leverage AI tooling
  effectively (this is a rapidly growing concern in hiring as of 2026).
- **Freelance clients or collaborators** browsing your GitHub between now and then.

The audience is *not* under your control or schedule. The portfolio sits on GitHub
and a personal site, waiting. This means:
- **Long shelf life matters more than speed.** A polished 12-week build beats a
  rushed 4-week build. There is no incentive to cut corners for time pressure.
- **The README is read more times than the code is.** Treat it as the primary
  artifact.
- **Date-stamping AI tooling carefully.** "Built with Claude Code (2026)" ages
  better than "Built with Sonnet 4.6" — the latter sounds dated within 6 months.
- **Avoid frothy AI-build marketing in the README.** Show the work, let it speak.
  The `docs/superpowers/` directory and the plan/design files do more credibility
  work than slogans.

What this audience needs to see in 60 seconds: a live demo URL, a README that
explains the gym metaphor and the AI-build workflow, and one screenshot or short
clip of the receipt-scan working on a real Indonesian struk.

## Premises (locked, agreed)

1. All 25 endpoints stay in scope. User's call — completeness as proof of stamina.
2. Showcase parts must be visible and deployed early — live URL by day 14 is the
   single biggest predictor of finishing.
3. Some thin demo surface (1-page frontend or polished Postman + curl recipes) is
   required so reviewers can verify the receipt-scan works without setting up the
   project locally.
4. Deploy + CI + tests come BEFORE feature breadth. A working Railway URL with one
   endpoint beats a complete plan that never deploys.

## Approaches Considered

### Approach A: Vertical Showcase First, Horizontal Fill After (chosen)
Foundation + deploy weeks 1–2. Three vertical slices through showcase features
(onboarding+budget engine, receipt scan+fatigue, weekly narrative) weeks 3–6.
Horizontal fill of remaining endpoints weeks 7–10. Thin frontend + polish weeks 11–12.

### Approach B: Execute the existing 16-phase plan sequentially
Phases 1–16 in order, deploy lands at the end (~14–16 weeks). Maximum rigor, lowest
cognitive overhead, highest motivation risk because no live URL for 3 months.

### Approach C: Frontend demo with fake JSON first, backend backfills
Polished Next.js demo week 1 with hard-coded responses, then progressively replace
fakes with real Go endpoints. Best public-facing story but dilutes the Go signal
that's the portfolio goal.
